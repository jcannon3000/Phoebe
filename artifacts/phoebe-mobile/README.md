# Phoebe Mobile

A Capacitor-wrapped iOS shell around the `mymonastery` web app. Zero changes to mymonastery source — the web app is the single source of truth for UI. Phoebe Mobile adds native shell features (push notifications, Sign in with Apple, haptics, biometrics, local notifications, safe-area insets, native share, keyboard behavior, deep links, native contacts) on top.

## Why Capacitor, not React Native?

- **Ship the existing web app immediately.** Every existing page, card, query, form works as-is.
- **Single source of truth.** Web changes land on mobile without a separate RN implementation.
- **Enough native capability to clear App Store Guideline 4.2** — see `src/native-shell.ts` for the full list.
- **Escape hatch.** Swift modules for widgets, Live Activities, and App Intents can coexist; Capacitor doesn't block them.

## Prerequisites

**On your machine:**
- macOS with Xcode 15+
- CocoaPods (`sudo gem install cocoapods`)
- Node 20+, pnpm 9+

**At Apple Developer Portal:**
- Apple Developer Program enrollment ($99/yr)
- App ID registered as `app.withphoebe.mobile`
- Capabilities enabled on that App ID: **Push Notifications**, **Sign in with Apple**, **Associated Domains**
- APNs Auth Key (.p8) — save the key, the key ID, and your team ID

**Environment variables** (Railway, for server-side push + SIWA):
- `APPLE_BUNDLE_ID=app.withphoebe.mobile` (defaults to this if unset)
- `APNS_KEY_P8` — the contents of the .p8 file (not a file path)
- `APNS_KEY_ID` — the 10-character key ID
- `APNS_TEAM_ID` — your Apple team ID

## First-time setup

```sh
# From repo root
pnpm install

# Build the web bundle + native shell into www/
pnpm --filter @workspace/phoebe-mobile run build

# Bootstrap the Xcode project (creates ios/ directory).
# You only do this once per repo.
cd artifacts/phoebe-mobile
npx cap add ios

# Sync Capacitor config + plugins into the Xcode project.
pnpm run cap:sync

# Open in Xcode
pnpm run cap:open
```

In Xcode:
1. Select the **App** target → **Signing & Capabilities** → set your **Team**.
2. Confirm **Bundle Identifier** is `app.withphoebe.mobile`.
3. Add capabilities (+ Capability):
   - **Push Notifications**
   - **Sign in with Apple**
   - **Associated Domains** → add `applinks:withphoebe.app`
   - **Keychain Sharing** (required by the biometrics plugin; group name `$(AppIdentifierPrefix)app.withphoebe.mobile`)
   - **App Groups** → add `group.app.withphoebe.mobile` (required for the widget to share data with the app)
4. Copy `assets/PrivacyInfo.xcprivacy` → `ios/App/App/PrivacyInfo.xcprivacy` and drag it into the App target in Xcode so it ships in the archive.
5. (Optional — Home Screen widget) File → New → Target → **Widget Extension**, name it `PhoebeWidget`, uncheck "Include Configuration App Intent". Replace the generated `PhoebeWidget.swift` with the contents of `assets/PhoebeWidget.swift`. Add the same **App Groups** capability to the widget target.
6. Select an iOS 17+ simulator or a physical device, ▶︎ Run.

## Iteration loop

Every time you change web app or native shell code:

```sh
# From repo root (or the phoebe-mobile directory):
pnpm --filter @workspace/phoebe-mobile run build
pnpm --filter @workspace/phoebe-mobile run cap:sync
# Then in Xcode, ⌘R to re-run on the sim.
```

For web-only changes you can skip `cap:sync` — the `www/` overlay is re-copied by `build:compose`.

## App Store submission checklist

Before hitting "Submit for Review":

- [ ] `apple-app-site-association` is served at `https://withphoebe.app/.well-known/apple-app-site-association` with the app's team ID + bundle ID, `Content-Type: application/json`, no redirects.
- [ ] `PrivacyInfo.xcprivacy` at `ios/App/App/PrivacyInfo.xcprivacy` declares the data Phoebe collects (email, name, device token for push) and that we don't track.
- [ ] App icon set: 1024×1024 base + all device sizes (use `npx @capacitor/assets generate`).
- [ ] Launch screen: matches the splash color `#091A10` so there's no flash.
- [ ] Sign in with Apple button is present and at LEAST as prominent as Google Sign-In on the login screen.
- [ ] The app does at least 3 things that require iOS to work (push, biometrics, local notifications, haptics, contacts) — enumerate them in the review notes.
- [ ] Privacy labels in App Store Connect match `PrivacyInfo.xcprivacy`.
- [ ] TestFlight internal testing has run for at least one full bell cycle (to verify push delivery).

## Native event bridge (for the web app)

The web app **never imports Capacitor**. Every native feature is invoked by dispatching a custom window event that the native shell listens for:

| Event (web → native) | Payload | Response events |
|---|---|---|
| `phoebe:request-push-permission` | — | `phoebe:push-ready`, `phoebe:push-denied`, `phoebe:push-error` |
| `phoebe:request-apple-signin` | — | `phoebe:apple-signin-ready`, `phoebe:apple-signin-error` |
| `phoebe:request-contacts` | — | `phoebe:contacts-ready { contacts }`, `phoebe:contacts-denied`, `phoebe:contacts-error` |
| `phoebe:schedule-bell` | `{ hourMin, bellId, title?, body? }` | `phoebe:bell-scheduled`, `phoebe:bell-denied`, `phoebe:bell-error` |
| `phoebe:cancel-bell` | `{ bellId }` | `phoebe:bell-cancelled` |
| `phoebe:share` | `{ title?, text?, url? }` | — |
| `phoebe:haptic` | `{ style: "light" \| "medium" \| "heavy" \| ... }` | — |

For the Home Screen widget, the web app writes `PhoebeNative.updateWidget({ bellTime, lectioStage, lectioPrompt, nextPracticeName })` whenever those values change. The shell mirrors that payload to a shared App Group suite that `PhoebeWidget.swift` reads on its next timeline refresh.

The `window.PhoebeNative.*` API is a convenience wrapper over these events — see the bottom of `src/native-shell.ts`.

On non-native builds (dev server, or if this shell accidentally loads in Safari), `PhoebeNative.isNative()` returns `false` and every other method is a no-op.

## Troubleshooting

- **"cap: command not found"** — run via pnpm scripts (`pnpm run cap:sync`) not bare `cap`.
- **Push token never arrives** — check that the simulator has push enabled (Device menu → Simulate Push) or use a real device; simulators only receive pushes on Xcode 14+ running macOS 13+.
- **Sign in with Apple prompts but nothing happens after** — confirm `/api/auth/apple/native` is live on the server (hit with curl) and that `APPLE_BUNDLE_ID` matches the app's bundle ID.
- **Face ID prompt appears every time on cold start** — that's intentional. The 5-minute grace window only applies to resumes from background.
- **Hot-reload doesn't reflect web changes** — run `pnpm run build:compose` (doesn't need a full web rebuild).
